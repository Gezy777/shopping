# GoMall - 云原生微服务电商项目

## 项目概述

GoMall 是一个基于 CloudWeGo 技术栈构建的云原生微服务电商平台，采用现代分布式架构设计，实现了完整的购物车、订单、支付等核心电商功能。这个项目可以作为学习微服务架构的实战参考，也可以作为快速启动电商业务的脚手架。

在当今日益复杂的互联网应用场景中，单体架构已经难以满足业务快速迭代和高可用的需求。GoMall 通过将电商核心业务拆分为多个独立的微服务，每个服务专注做好一件事，实现了服务的独立部署、扩展和维护，降低了系统各部分之间的耦合度，提高了整体的灵活性和可靠性。

### 技术栈

| 类别 | 技术 | 说明 |
|------|------|------|
| **前端网关** | Hertz | 字节跳动开源的高性能 HTTP 框架 |
| **RPC 框架** | Kitex | 字节跳动开源的高性能 RPC 框架 |
| **服务注册与发现** | Consul | HashiCorp 推出的服务网格解决方案 |
| **数据库** | MySQL + GORM | 关系型数据库 + Go ORM 框架 |
| **缓存** | Redis | 高性能内存数据库 |
| **消息队列** | NATS | 轻量级、高性能的消息系统 |
| **AI 集成** | Ollama (Eino 组件) | 本地大模型推理框架 |
| **基础设施** | Docker Compose, Kubernetes | 容器化部署方案 |

## 架构设计

### 设计理念

GoMall 的架构设计遵循了微服务架构的核心原则：将复杂的业务拆分为多个松耦合的服务，每个服务拥有独立的数据库，通过轻量级的通信协议进行交互。这种设计带来以下优势：

- **独立部署**：每个服务可以独立部署，不影响其他服务
- **技术多样性**：不同服务可以根据需求选择不同的技术栈
- **故障隔离**：一个服务的故障不会导致整个系统崩溃
- **弹性扩展**：根据负载情况，对特定服务进行水平扩展

## 服务详解

### 1. Frontend (前端网关)

Frontend 是整个系统的门面，类似于传统架构中的 API Gateway 或 Nginx。它使用 Hertz 框架构建，负责接收所有来自浏览器的 HTTP 请求，并将这些请求路由到对应的后端微服务。

**技术框架**: Hertz + HTML 模板

**为什么选择 Hertz？** Hertz 是字节跳动内部使用多年的 HTTP 框架，经过了大量生产环境的验证。它采用异步设计，具有极高的吞吐量和极低的延迟。同时 Hertz 提供了丰富的中间件生态，包括 CORS、Session、压缩、日志等功能，开发者可以像搭积木一样组合使用这些中间件来满足不同的需求。

**核心功能**:

Frontend 服务不仅仅是一个简单的路由转发器，它还承担了页面渲染的职责。对于电商平台来说，需要展示大量的动态页面，如商品详情页、购物车页、结算页等。Frontend 使用 Go 的 HTML 模板引擎来渲染这些页面，将业务数据填充到模板中生成最终的 HTML 页面返回给用户。

此外，Frontend 还负责管理用户会话。它使用 Redis 来存储用户的 Session 数据，包括用户的登录状态、购物车内容等。通过 Session 中间件，开发者可以方便地获取当前登录用户的信息，实现用户认证和授权功能。

**主要路由**:

| 路径 | 功能 | 说明 |
|------|------|------|
| `GET /` | 首页 | 展示商品列表和分类 |
| `GET /about` | 关于页面 | 展示平台介绍信息 |
| `GET /sign-in` | 登录页 | 用户登录表单页面 |
| `GET /sign-up` | 注册页 | 用户注册表单页面 |
| `/product/*` | 商品相关 | 商品搜索、详情页路由 |
| `/cart/*` | 购物车相关 | 购物车查看、操作路由 |
| `/checkout/*` | 结账相关 | 结算页面路由 |
| `/order/*` | 订单相关 | 订单列表、订单详情路由 |

### 2. Product (商品服务)

商品服务是电商平台最核心的服务之一，它管理着平台上的所有商品信息。一个电商平台的本质就是"卖商品"，无论是购物车、订单还是结算，都离不开商品信息。

**技术框架**: Kitex + GORM + MySQL

**为什么使用 GORM？** GORM 是一个功能完善的 Go ORM 库，它的 API 设计优雅，链式调用风格让代码读起来非常自然。同时 GORM 提供了自动迁移、预加载、事务支持等高级功能，可以大大简化数据库操作的代码。

**核心接口**:

Product 服务的接口设计遵循了"最小化接口"的原则，只提供业务真正需要的方法，不多也不少：

- `ListProducts` - 分页获取商品列表，这是商品浏览的核心接口，需要支持按分类筛选和分页
- `GetProduct` - 获取单个商品详情，用于商品详情页的展示
- `GetProductByName` - 按名称精确查询商品，主要用于购物车中商品的验证
- `SearchProducts` - 搜索商品，通过关键词匹配商品名称和描述

**数据模型**:

商品模型包含了商品的核心属性：商品 ID、名称、描述、图片 URL、价格、所属分类等。其中分类信息使用 repeated string 类型存储，这意味着一个商品可以属于多个分类，比如一件衣服可以同时属于"女装"和"夏季新品"两个分类。

### 3. Cart (购物车服务)

购物车服务负责管理用户的购物车内容。与其他服务不同，购物车服务使用 Redis 作为存储，这是出于性能考虑。购物车是一个读写非常频繁的数据结构，用户可能频繁地添加、删除、修改商品，如果每次操作都访问数据库，会给数据库带来很大的压力。而 Redis 是内存数据库，读写速度极快，非常适合存储这类热点数据。

**技术框架**: Kitex + Redis

**为什么使用 Redis？** Redis 提供了丰富的数据结构支持，购物车服务使用 Redis 的 Hash 结构来存储购物车内容，每个用户的购物车是一个独立的 Hash，Key 是用户 ID，Field 是商品 ID，Value 是商品数量。这种设计使得查询某个用户的购物车、添加商品、删除商品等操作都可以在 O(1) 或 O(n) 的时间复杂度内完成。

**核心接口**:

- `AddItem` - 添加商品到购物车，如果商品已存在则增加数量，否则新建条目
- `GetCart` - 获取用户购物车中的所有商品
- `EmptyCart` - 清空购物车，通常在用户完成订单后调用

### 4. Order (订单服务)

订单服务是电商平台中最复杂的服务之一，它负责处理订单的创建、查询和管理。一个订单涉及到商品的快照、价格的计算、地址信息、支付状态等多个方面，需要在数据库层面保证数据的一致性。

**技术框架**: Kitex + GORM + MySQL

**事务处理**: 订单服务大量使用数据库事务来保证数据一致性。例如在创建订单时，需要同时创建订单主记录和订单商品明细记录，这两个操作必须在同一个事务中完成，否则就会出现数据不一致的情况。GORM 的事务 API 非常简洁，可以很方便地实现这种需求。

**核心接口**:

- `PlaceOrder` - 创建订单，这是最复杂的接口，需要计算订单金额、保存收货地址、记录订单商品明细
- `ListOrder` - 查询用户的订单列表，需要支持分页和排序
- `MarkOrderPaid` - 标记订单已支付，通常由支付服务在扣款成功后调用

**订单状态机**: 订单有多个状态，包括待支付、已支付、已发货、已完成、已取消等。状态之间的转换需要遵循一定的规则，例如只有待支付的订单才能取消，只有已支付的订单才能发货。

### 5. Checkout (结账服务)

结账服务是整个购物流程的核心，它像一个协调者一样，串联起购物车、商品、订单、支付、邮件等多个服务。结账服务的设计理念是"一步完成"，用户只需要调用一个接口，就能完成从查看购物车到下单支付的全部流程。

**技术框架**: Kitex + NATS

**业务流程**:

当用户点击"结算"按钮时，Checkout 服务会按照以下步骤执行：

第一步，获取购物车内容。调用 Cart 服务的 GetCart 接口，获取用户购物车中的所有商品。如果购物车为空，直接返回错误。

第二步，遍历购物车商品，查询每个商品的最新价格。这一步很重要，因为商品价格可能随时变动，用户购物车中保存的可能是很久以前添加的商品，价格可能已经调整。

第三步，计算订单总金额。将每个商品的价格乘以数量，然后累加得到订单总金额。

第四步，创建订单。调用 Order 服务的 PlaceOrder 接口，传入收货地址、商品明细等信息，创建订单记录。

第五步，调用支付服务扣款。这一步是真正的"付费"操作，需要用户输入信用卡信息。

第六步，清空购物车。无论支付是否成功，购物车都应该被清空，避免用户重复下单。

第七步，发送邮件通知。这是通过 NATS 消息队列异步执行的，Checkout 服务将邮件内容发布到 NATS 的 "email" 主题，由 Email 服务异步处理。这种异步设计可以提高系统的响应速度，用户不需要等待邮件发送完成。

**为什么使用 NATS？** NATS 是一个轻量级、高性能的消息系统，它采用发布-订阅模式，非常适合这种跨服务的异步通信场景。与传统的消息队列（如 RabbitMQ、Kafka）相比，NATS 配置更简单，性能更高，而且支持多种消息模式。

### 6. Payment (支付服务)

支付服务负责处理支付扣款逻辑。在这个简化版的电商平台中，支付服务只是模拟了扣款过程，并没有真正对接银行或第三方支付渠道。在生产环境中，这里需要集成真实的支付网关。

**核心接口**:

- `Charge` - 扣款操作，接受金额、信用卡信息、订单 ID 等参数，返回交易 ID

**安全性考虑**: 在真实的支付系统中，信用卡等敏感信息需要加密存储，并且不能记录在日志中。支付服务的实现应该遵循 PCI-DSS（支付卡行业数据安全标准）的要求。

### 7. Email (邮件服务)

邮件服务负责向用户发送通知邮件。在 GoMall 中，邮件服务通过 NATS 消息队列接收邮件发送请求，实现了与业务逻辑的解耦。

**技术框架**: Kitex + NATS

**订阅机制**: Email 服务订阅了 NATS 的 "email" 主题，当有邮件发送请求时，NATS 会将消息推送给 Email 服务。这种设计的好处是邮件发送不会阻塞主流程，即使邮件服务暂时不可用，消息也会在 NATS 中保留，等服务恢复后继续处理。

**应用场景**:

- 用户创建订单后，发送订单确认邮件
- 订单支付成功后，发送支付成功邮件
- 订单发货后，发送物流通知邮件
- 用户忘记密码时，发送密码重置链接

### 8. User (用户服务)

用户服务负责处理用户的注册和登录，这是电商平台的基础功能。没有用户系统，就无法管理订单、无法维护用户偏好，因此用户服务是整个平台的基础。

**技术框架**: Kitex + MySQL

**核心接口**:

- `Register` - 用户注册，创建新的用户账号
- `Login` - 用户登录，验证用户凭据并返回用户 ID

**密码安全**: 在实际的系统中，用户密码不能明文存储，需要使用 bcrypt 或 argon2 等算法进行哈希处理。即使数据库被拖库，攻击者也无法直接获取用户的明文密码。

## 业务流详解

### 购物流程

```
用户浏览 → 商品搜索 → 查看商品详情
              ↓
         添加购物车
              ↓
         购物车结算 → Checkout 服务
              ↓
    ┌─────────┼─────────┐
    ↓         ↓         ↓
 获取商品  创建订单  调用支付
    ↓         ↓         ↓
 计算价格  保存订单  扣款处理
              ↓
         清空购物车
              ↓
         发送邮件 (NATS)
```

整个购物流程从用户浏览商品开始。用户可以在首页看到商品列表，也可以通过搜索功能查找感兴趣的商品。当用户找到心仪的商品后，点击查看商品详情，了解更多关于商品的描述、价格、图片等信息。

如果用户决定购买，可以将商品加入购物车。购物车允许用户同时添加多个商品，并可以随时修改数量或删除商品。这个过程完全是在客户端和 Cart 服务之间进行的，不需要其他服务参与。

当用户准备下单时，点击"结算"按钮，进入结算页面。在这个页面，用户需要确认收货地址、选择支付方式等。确认后，Frontend 调用 Checkout 服务的 Checkout 接口，开始结账流程。

Checkout 服务作为协调者，会依次调用 Cart 服务获取商品、Product 服务获取商品价格、Order 服务创建订单、Payment 服务完成支付。整个过程是同步的，用户需要等待所有步骤完成后才能看到结果。

结账完成后，Checkout 服务会通过 NATS 发送一封邮件通知，告知用户订单已创建。用户可以在订单页面查看订单详情和物流信息。

### 服务间调用关系

```
Frontend
   │
   ├──────────────────────────────┐
   │                              │
   ▼                              ▼
Product ◄───────────────────── Cart
   │                              │
   │                              │
   └──────────┬───────────────────┘
              │
              ▼
         Checkout
              │
      ┌───────┼───────┐
      │       │       │
      ▼       ▼       ▼
   Order   Payment   Email
              │
              ▼
           Cart (EmptyCart)
```

从调用关系图可以看出，Frontend 是所有调用的入口，它直接调用 Product 和 Cart 服务。Checkout 服务是业务逻辑的核心，它调用的服务最多：Product（获取商品信息）、Order（创建订单）、Payment（支付扣款）、Cart（清空购物车）、Email（发送邮件）。

这种调用链的设计使得服务之间的依赖关系清晰可见。在架构演进的过程中，如果某个服务的功能变得太复杂，可以继续拆分。例如，Checkout 服务未来可以拆分为 Checkout 和 Order Processing 两个服务，Order 服务也可以拆分为 Order 和 Inventory（库存）服务。

## 项目特点

### 1. 现代化微服务架构

GoMall 采用了当前业界主流的微服务架构设计理念，将复杂的电商业务拆分为多个独立的微服务。这种架构带来的好处是显而易见的：

**独立部署和扩展**: 每个微服务可以独立部署，不影响其他服务。当某个服务的负载增加时，只需要扩展该服务，而不需要扩展整个系统。例如，在促销活动中，商品服务和购物车服务可能会承受比平时高几倍的流量，这时可以只扩展这两个服务，而其他服务保持原样。

**技术多样性**: 不同服务可以根据其特点选择最合适的技术栈。例如，购物车服务选择 Redis 是因为它需要极高的读写性能；商品和订单服务选择 MySQL 是因为它们需要事务支持和复杂查询能力。

**故障隔离**: 良好的微服务架构应该具备故障隔离能力。当 Email 服务不可用时，不应该影响用户下单。因为邮件发送是通过 NATS 异步处理的，即使 Email 服务暂时不可用，消息也会在队列中保留。

### 2. 完整的技术栈覆盖

GoMall 涉及的技术栈非常全面，涵盖了微服务开发的各个方面：

**RPC 通信**: Kitex 框架提供了连接池、超时控制、熔断、重试等高级特性。连接池可以复用 TCP 连接，减少连接建立的开销；超时控制可以防止请求无限等待；熔断机制可以在下游服务不可用时快速失败，避免资源耗尽；重试机制可以在网络抖动时自动重试，提高系统的健壮性。

**HTTP 服务**: Hertz 框架的异步设计使其具有极高的性能。配合中间件机制，可以方便地添加 CORS 支持、请求压缩、访问日志、性能分析等功能。

**服务发现**: Consul 不仅提供了服务注册功能，还提供了健康检查和 DNS 发现。通过健康检查，Consul 可以自动剔除不可用的服务实例；通过 DNS 发现，客户端可以通过服务名而不是 IP 地址来访问服务。

**消息队列**: NATS 的发布-订阅模式非常适合跨服务的异步通信。与传统的消息队列相比，NATS 更加轻量级，配置更简单，性能也更高。

### 3. 清晰的代码分层

GoMall 的每个服务都遵循了清晰的分层架构：

```
┌─────────────┐
│   Handler   │  负责接收请求、参数校验、调用 Service、返回响应
├─────────────┤
│  Service    │  负责实现业务逻辑，编排多个 DAL 操作
├─────────────┤
│    DAL      │  负责与数据库或缓存交互，实现数据持久化
├─────────────┤
│   Model     │  定义数据结构和业务模型
└─────────────┘
```

Handler 层是请求的入口，它负责解析请求参数、调用相应的 Service 方法、并将结果封装成响应格式返回。Handler 层不应该包含业务逻辑，它只是一个转发层。

Service 层是业务逻辑的核心，它定义了"做什么"。一个 Service 通常对应一个业务操作，如创建订单、添加购物车等。Service 会调用多个 DAL 方法来完成业务逻辑，并处理事务。

DAL（Data Access Layer）层负责与底层存储交互，它定义了"怎么做"。DAL 层封装了所有的数据库操作，向上层提供简洁的接口。

Model 层定义了数据结构和业务模型，它们是整个系统的基础。

### 4. 完善的测试支持

每个 Service 层都有对应的 `*_test.go` 文件，cwgo 工具会自动生成单元测试的骨架代码，开发者只需要填充具体的测试逻辑即可。这种开箱即用的测试支持，有助于保证代码质量。

测试应该覆盖：
- 正常业务流程
- 边界条件和异常输入
- 错误处理逻辑
- 并发场景

### 5. 基础设施完善

GoMall 提供了完整的开发和部署支持：

**Docker Compose**: 在开发环境中，可以使用 docker-compose 一键启动所有依赖的服务，不需要手动安装和配置 MySQL、Redis 等软件。

**Kubernetes**: k8s_demo 目录下提供了 Kubernetes 部署配置，包括 Deployment、Service 等资源清单，可以直接部署到 K8s 集群。

**Makefile**: 自动化构建脚本，将常用的命令封装成 make targets，减少重复输入。

## 开发指南

### 添加新的微服务

当需要添加一个新的微服务时，可以按照以下步骤进行：

**第一步，定义 IDL**

在 idl/ 目录下创建新的 .proto 文件，定义服务的接口和数据结构。例如，创建 inventory.proto 来管理商品库存：

```protobuf
syntax = "proto3";

package inventory;

option go_package = "/inventory";

service InventoryService {
  rpc GetStock(GetStockReq) returns (GetStockResp) {}
  rpc DeductStock(DeductStockReq) returns (DeductStockResp) {}
}

message GetStockReq {
  uint32 product_id = 1;
}

message GetStockResp {
  int32 stock = 1;
}

message DeductStockReq {
  uint32 product_id = 1;
  int32 quantity = 2;
}

message DeductStockResp {}
```

**第二步，生成代码**

使用 cwgo 工具生成服务端代码：

```bash
cwgo server --type RPC \
  --service inventory \
  --module github.com/cloudwego/biz-demo/gomall/app/inventory \
  --pass "use github.com/cloudwego/biz-demo/gomall/rpc_gen" \
  -I ../../idl \
  --idl ../../idl/inventory.proto
```

**第三步，实现业务逻辑**

生成的代码只包含了接口和骨架，需要在 Service 层实现具体的业务逻辑。

### 服务间调用

当一个服务需要调用另一个服务时，需要使用该服务的 RPC 客户端。以 Frontend 调用 Product 服务为例：

**初始化客户端**：

```go
// 在 infra/rpc/init.go 中初始化
ProductClient = product.NewClient("product",
    client.WithHostPorts("127.0.0.1:8080"),
    client.WithMiddleware(client.Middleware),
)
```

**调用服务**：

```go
resp, err := ProductClient.GetProduct(ctx, &product.GetProductReq{
    Id: 1,
})
if err != nil {
    return err
}
```

## 配置说明

GoMall 采用分层配置管理，每个服务都有独立的环境配置。

### 目录结构

每个服务的配置目录结构如下：

```
app/<service>/
├── conf/
│   ├── conf.go          # 配置加载器
│   ├── dev/
│   │   └── conf.yaml    # 开发环境配置
│   ├── test/
│   │   └── conf.yaml    # 测试环境配置
│   └── online/
│       └── conf.yaml    # 生产环境配置
└── .env                # 环境变量文件
```

### 主要配置项

**服务配置**：
- `kitex.address` - RPC 服务监听地址，格式为 "IP:Port"
- `hertz.address` - HTTP 服务监听地址

**注册中心**：
- `registry.registryAddress` - Consul 服务地址，用于服务注册和发现

**数据库**：
- `mysql.*` - MySQL 连接配置，包括地址、用户名、密码、数据库名等

**缓存**：
- `redis.*` - Redis 连接配置

**日志**：
- `kitex.logLevel` - Kitex 日志级别
- `kitex.logFileName` - 日志文件名
- `kitex.logMaxSize` - 日志文件最大大小
- `kitex.logMaxBackups` - 保留的日志文件数量
- `kitex.logMaxAge` - 日志文件保留天数

### 环境切换

通过设置环境变量或修改 .env 文件，可以切换不同的配置环境：

```bash
export GO_ENV=online
```
